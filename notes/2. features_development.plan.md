
# Phased feature development and release cadence

## Phase sizing (Goldilocks chunks)

Each phase is **one vertical slice**: API + DB + minimal UI—not a single endpoint, not micro-tasks. Order respects dependencies (board ACL needs users first).

---

### Phase 1 — Foundation, auth, and “logged-in shell”

**Build**

- Repo layout. `docker compose`, each service (frontend, API, db, E2E tests) has its' own container ([TECH_STACK.md](TECH_STACK.md)).
- Users: register + login; JWT (or session) issuance; MFA, password hashing; protected API routes and Vue **route guards** (redirect unauthenticated users).
- Minimal UI: login/register pages + empty “home” or placeholder after login.

**Out of scope for this phase:** boards.

#### User acceptance criteria

- Given a new user is on the registration page, when they submit a valid email and password, then an account is created and they can proceed to login.
- Given a user enters an already registered email during registration, when they submit the form, then they see a clear validation error and no duplicate account is created.
- Given a registered user is on the login page, when they provide valid credentials (and valid MFA code if MFA is enabled), then they receive authenticated access.
- Given an unauthenticated user tries to open a protected route, when route guard is evaluated, then the user is redirected to login.
- Given an authenticated user opens the app, when authentication is valid, then they can access the minimal post-login home/placeholder page.

---

### Phase 2 — Boards and Trello-like permissions

**Build**

- **Board** CRUD; **board membership**: owner (creator) + invited members; non-members cannot read/write board (simplify to “owner + members with same capability” unless you want viewer/editor roles—pick one and document in README).
- API: invite/add user to board by email or username; list boards visible to user.
- UI: board list, create board, open board shell (lists area can be empty).

---

### Phase 3 — Lists and tasks (core task model)

**Build**

- **Lists** on a board; **tasks** in a list with **title** + **description**; reorder/move task between lists; edit/delete task; ensure all operations respect board membership from Phase 2.

---

### Phase 4 — Tags, assignee, and comments

**Build**

- Task tags: exactly the three enum values from [notes/app_design.md](notes/app_design.md) (`Important`, `Needs Definition`, `Unscoped`); assign **one** member per task (must be a board member).
- **Comments** on tasks (author, timestamp).

---

### Phase 5 — Profile, due date ~~, and MFA~~

**Build**

- **Profile** edit: username and password change (re-login or token refresh as you prefer).
- Optional **date + time** on task; validation and display; timezone policy documented (e.g. store UTC, show local).
- ~~**MFA:** TOTP or similar; backup codes; login flow change. Keep Phase 1 auth simple so MFA is additive.~~

---

## Cross-cutting work (once, then maintained)

- **DB migrations** on each deploy (all envs).
- **README:** API base URL pattern, auth header, and **permission model** (required by [notes/app_design.md](notes/app_design.md)).
- **Stage seeding:** script or migration seed for E2E users/boards so E2E does not depend on manual setup.
- **GitHub Actions:** PR = fast checks; **`stage`** = deploy + E2E against stage URL; **`main`** = deploy + smoke ([TECH_STACK.md](TECH_STACK.md)).
